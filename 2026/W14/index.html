daily-fetch week rollover bug fixed after Sunday event loss

A critical bug in the daily-fetch cron caused Sunday events to be missed due to ISO week miscalculation, prompting a week of debugging, fixes, and collateral improvements.

This week started with a routine maintenance mode but quickly pivoted when I discovered that Sunday's GitHub events were missing from the weekly reports. The root cause was a subtle bug in the daily-fetch script's week calculation logic, which used the previous ISO week instead of the current one when running mid-week.

After identifying the issue in buildWeeklyRange and getWeekId, I implemented buildCurrentWeekRange and getCurrentWeekId to ensure daily-fetch always collects the current week's events. This fix triggered a cascade of related improvements: better PR fetch error handling, exclusion of private repos from searches, and enhanced documentation.

By week's end, the main repository saw 19 PRs merged, test coverage jumped from 40% to 97%, and I set up Dependabot to automate future dependency updates. The burst of activity on Tuesday and Thursday, with 161 and 283 commits respectively, reflected the intensity of the debugging and subsequent refactoring.

The Sunday event loss mystery
On Monday, I noticed that Sunday's events were absent from the weekly report. After reproducing the issue, I found that daily-fetch's cron at midnight on Monday used buildWeeklyRange which targeted the previous ISO week (W14), so Sunday's events (still in W14) were never collected. The fix involved creating buildCurrentWeekRange and getCurrentWeekId to always target the current week, ensuring no events are missed.
